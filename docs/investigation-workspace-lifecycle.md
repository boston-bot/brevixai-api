# Investigation Workspace Lifecycle

## Overview

The investigation workspace is a structured layer on top of `audit_cases` that gives analysts a dedicated workflow for managing active investigations. Every investigation is tied to an `audit_case`, which is typically created when a user approves a `case_recommendation` generated by the risk scoring pipeline.

Agents have no direct access to investigations. Agents may surface recommendations; humans approve them and open investigations.

---

## Lifecycle

```
CaseRecommendation (pending_review)
        │
        │  User approves via POST /api/case-recommendations/{id}/approve
        ▼
AuditCase created (status: open)
        │  investigation_status: open (default)
        │  investigation_priority: medium (default)
        │  InvestigationActivityEvent: case_created
        │
        │  POST /api/investigations/{id}/assign
        ▼
investigation_assigned_user_id set
        │  InvestigationActivityEvent: assigned
        │
        │  POST /api/investigations/{id}/status
        ▼
investigation_status: in_review → escalated → resolved → archived
        │  InvestigationActivityEvent: status_changed (each transition)
        │
        │  POST /api/investigations/{id}/notes
        ▼
investigation_notes updated
           InvestigationActivityEvent: notes_added
        │
        │  User requests POST /api/investigations/{id}/package-manifest
        ▼
sanitized package manifest returned
           InvestigationActivityEvent: package_manifest_generated
```

## Investigation Status Values

| Status       | Meaning                                              |
|--------------|------------------------------------------------------|
| `open`       | Case exists, investigation not yet started           |
| `in_review`  | Analyst is actively reviewing the case               |
| `escalated`  | Elevated for senior review or external action        |
| `resolved`   | Investigation concluded                              |
| `archived`   | Closed and archived; no further action expected      |

---

## Investigation Priority Values

| Priority   | Use when                                              |
|------------|-------------------------------------------------------|
| `critical` | Immediate risk of financial harm or fraud in progress |
| `high`     | Strong signals requiring prompt review                |
| `medium`   | Moderate signals; standard review queue (default)     |
| `low`      | Informational; review when capacity allows            |

## Activity Timeline Events

All writes to the investigation workspace produce an `investigation_activity_event` record. The timeline is append-only and never modified.

| Event Type              | Triggered by                                      |
|-------------------------|---------------------------------------------------|
| `case_created`          | Case recommendation approved by user              |
| `assigned`              | POST /api/investigations/{id}/assign              |
| `status_changed`        | POST /api/investigations/{id}/status              |
| `notes_added`           | POST /api/investigations/{id}/notes               |
| `evidence_linked`       | Future: evidence linking endpoint                 |
| `recommendation_approved` | Future: additional recommendation approvals     |
| `report_generated`      | User generated an investigation report export     |
| `package_manifest_generated` | User generated a sanitized export manifest |

### Actor Types

| Type     | When used                                              |
|----------|--------------------------------------------------------|
| `user`   | All user-initiated workspace actions                   |
| `system` | Automated processes (expiration, scheduled checks)     |
| `agent`  | Agent-originated events (agents cannot write directly) |

**Agents cannot write investigation activity events directly.** Any agent-generated events must be proxied through a user-authorized workflow.

### Sensitive Data Exclusion

`event_metadata` is sanitized before storage. The following keys are stripped at any nesting level:

- `evidence`
- `supporting_evidence`
- `raw_evidence`
- `transaction_details`
- `raw_payload`

Raw evidence payloads are never stored in the activity timeline. Evidence summaries (domain names, counts) are permitted.

## API Endpoints

All endpoints require `auth:sanctum` and enforce company-scoped access. Users can only access investigations for their own company.

| Method | Path                               | Description                          |
|--------|------------------------------------|--------------------------------------|
| GET    | `/api/investigations`              | List investigations with filters     |
| GET    | `/api/investigations/{id}`         | Full investigation detail            |
| POST   | `/api/investigations/{id}/assign`  | Assign to a user (must be same co.)  |
| POST   | `/api/investigations/{id}/status`  | Update investigation_status          |
| POST   | `/api/investigations/{id}/notes`   | Add or update investigation notes    |
| GET    | `/api/investigations/{id}/reports` | List report export records           |
| POST   | `/api/investigations/{id}/reports` | Generate a JSON or PDF report export |
| POST   | `/api/investigations/{id}/package-manifest` | Generate a sanitized package manifest |

### List Query Parameters

| Parameter               | Type   | Values                                                    |
|-------------------------|--------|-----------------------------------------------------------|
| `investigation_status`  | string | `all`, `open`, `in_review`, `escalated`, `resolved`, `archived` |
| `investigation_priority`| string | `critical`, `high`, `medium`, `low`                       |
| `assigned_to`           | uuid   | Filter by assigned user ID                               |
| `limit`                 | int    | 1–100 (default 50)                                       |
| `offset`                | int    | Pagination offset (default 0)                            |

## Package Manifest Lifecycle

The package manifest is an on-demand JSON summary of the materials that would be referenced in an investigation export package. It is intended for human review before sharing or archiving investigation materials. The manifest is not the package itself, does not create downloadable archives, and is not generated by background jobs or agents.

### Request

```http
POST /api/investigations/{id}/package-manifest
```

```json
{
  "format": "json"
}
```

### Response Shape

```json
{
  "manifest": {
    "investigation_id": "uuid",
    "generated_at": "ISO8601",
    "generated_by_user_id": "uuid",
    "included_sections": [
      "report_exports",
      "evidence_items",
      "linked_alerts",
      "linked_recommendations",
      "activity_events",
      "notes",
      "disclaimer"
    ],
    "included_counts": {
      "report_exports": 1,
      "evidence_items": 3,
      "linked_alerts": 2,
      "linked_recommendations": 2,
      "activity_events": 5,
      "notes": 1
    },
    "report_exports": [],
    "evidence_items": [],
    "linked_alerts": [],
    "linked_recommendations": [],
    "activity_events": [],
    "notes": [],
    "disclaimer": "This package manifest summarizes included investigation materials and review activity. It is not a legal conclusion or proof of fraud."
  }
}
```

### Sanitization Rules

- Package manifest generation is user-triggered only through authenticated `auth:sanctum` investigation routes.
- Agents cannot generate package manifests. No internal agent tool route exposes package manifest generation.
- The manifest is not stored. Only a `package_manifest_generated` activity event is recorded.
- Report export entries are references only: export id, generator, format, filename, report hash, generation time, and safe count fields.
- Evidence items include ledger references and investigator-facing title/summary fields, but never `metadata`.
- Activity events include event references and summaries, but never `event_metadata`.
- Linked alerts omit raw `detail` and `evidence`.
- Linked recommendations omit raw `evidence`, `source_rule_ids`, and review notes.
- Investigation notes are represented by reference, summary, character count, and timestamp. The note body is not returned in the manifest.

The manifest supports review and chain-of-custody workflows. It is not a legal conclusion, proof of fraud, or a substitute for reviewing the underlying materials.

## Authorization Rules

- Users can only list and view investigations scoped to their `company_id`.
- Users can only assign investigations to users within the same company.
- There is no cross-company access; a 404 is returned for any resource the user's company does not own.
- Agents cannot call investigation endpoints. The `agent.tool` middleware does not expose investigation routes.

## Database Schema

### `audit_cases` — Investigation Columns (added in Phase 4.1)

| Column                          | Type      | Default    | Description                             |
|---------------------------------|-----------|------------|-----------------------------------------|
| `investigation_status`          | text      | `open`     | Workspace-level investigation status    |
| `investigation_assigned_user_id`| uuid FK   | null       | Assigned investigator (nulls on delete) |
| `investigation_priority`        | text      | `medium`   | Triage priority                         |
| `investigation_summary`         | text      | null       | High-level synthesis written by analyst |
| `investigation_notes`           | text      | null       | Working notes (mutable)                 |
| `last_activity_at`              | timestamptz | null     | Updated on every activity event         |
| `investigation_metadata`        | jsonb     | null       | Extensible structured workspace data    |

### `investigation_activity_events`

| Column           | Type        | Description                                      |
|------------------|-------------|--------------------------------------------------|
| `id`             | uuid PK     | Event identifier                                 |
| `audit_case_id`  | uuid FK     | Parent investigation (cascades on delete)        |
| `company_id`     | uuid FK     | Denormalized for query efficiency                |
| `event_type`     | text        | See event type table above                       |
| `actor_type`     | text        | `user`, `system`, or `agent`                     |
| `actor_id`       | uuid        | Nullable; null for system events                 |
| `event_summary`  | text        | Human-readable description of the event          |
| `event_metadata` | jsonb       | Sanitized structured context (no raw evidence)   |
| `created_at`     | timestamptz | Set at insert; immutable                         |
